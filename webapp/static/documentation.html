01 — Workflow

End-to-End Transaction Flow

Merchant Initiates Transaction
Merchant enters the payment amount in the POS dashboard and submits a payment request. The Flask backend receives this and assigns a unique transaction_id.
POS Dashboard → Flask API
Session & OTP Generated
Server creates a cryptographically random 4-digit challenge OTP and binds it to the transaction session. OTP is stored in-memory, scoped to this exact transaction ID.
In-memory OTP store
QR Payload Constructed
The customer verification URL (containing the transaction ID) is encoded into a QR code matrix using the qrcode library. The result is a binary pixel matrix.
qrcode + NumPy matrix
Visual Cryptography Split
The QR matrix is split using a (2,2) visual secret sharing scheme. Two independent shares are generated — neither share alone reveals any readable information about the QR code.
Visual cryptography (2,2) threshold
MQTT Publish to ESP32
The QR pixel data and OTP challenge code are serialized and published to a specific MQTT topic via the Paho broker. The ESP32 device, subscribed to this topic, receives the message in real-time.
Paho MQTT → broker → ESP32
Trusted Hardware Displays OTP
The ESP32 parses the MQTT payload and renders the 4-digit OTP on its OLED screen. This is the critical out-of-band factor — the challenge is shown on a trusted physical device the attacker cannot read remotely.
ESP32 + OLED (out-of-band)
Customer Scans QR & Enters OTP
Customer scans the merchant's displayed QR code and opens the verification page. They physically read the OTP from the OLED screen and type it into the web form — proving physical presence.
Human verification step
Backend Validates & Completes
Flask receives the OTP, checks it against the session-scoped store, and immediately invalidates it. If valid, the transaction is marked complete and a success response is returned.
OTP invalidated after first use
Shares Reconstructed (Cryptographic Proof)
Optionally, the two QR shares are recombined via XOR overlay to reconstruct the original QR for audit/logging purposes — demonstrating the (2,2) threshold is satisfied.
XOR share reconstruction
Why visual cryptography? A screen-capture attack or MITM proxy cannot intercept the OTP because it is delivered exclusively via a physical OLED display that the attacker has no network access to. Both the QR and the OTP must be colluded to complete payment.
02 — Architecture

Core System Layers

Frontend Layer

  • Landing page for product overview and merchant entry points
  • POS merchant interface: amount entry, QR display, and challenge dispatch
  • Customer verification flow: OTP input, real-time status polling
  • Static HTML/CSS/JS served directly by Flask

Backend Layer (Flask)

  • REST endpoints: /start, /verify, /status
  • In-memory transaction and OTP state management
  • QR generation pipeline and MQTT publish orchestration
  • Session scoping prevents OTP cross-transaction leakage

Cryptography Layer

  • (2,2) visual secret sharing — both shares required for reconstruction
  • Single share reveals zero readable information (information-theoretic)
  • Anti-capture: screen recordings of share alone are useless
  • NumPy XOR operations for share generation and reconstruction

IoT Layer (ESP32 + MQTT)

  • ESP32 subscribes to MQTT topic on boot
  • Receives JSON payload containing OTP on transaction start
  • OLED renders OTP as the trusted out-of-band display
  • Physically decouples approval from potentially compromised UI

04 — Security

Security Guarantees

Possession Factor

Flow requires physical access to the registered ESP32 hardware — no remote-only attack can succeed.

Ephemeral OTP

Time-sensitive code drastically lowers replay and delayed interception risk. Expires after single use.

Session Scoping

OTP is cryptographically bound to a specific transaction_id — invalid in any other context.

One-Time Semantics

OTP is invalidated immediately after successful verification, defeating any replay attempt.

Encrypted Transport

MQTT channel should run with TLS for encrypted broker-device communication in production.

Phishing Resistance

Relay attacks fail — attacker cannot read the OLED output in real-time without physical proximity.

06 — Roadmap

Limitations & Future Upgrades

Current Limitations
  • In-memory OTP/session store does not survive server restarts
  • Single-instance architecture limits horizontal scaling
  • Demo-oriented broker/topic setup needs hardening for multi-tenancy
  • Observability, auditing, and logging are minimal at this stage
Planned Upgrades
  • Redis / PostgreSQL for persistent OTP and session state
  • Mutual TLS + cert pinning to strengthen broker/device trust
  • Hardware attestation — device-bound key verification
  • Risk scoring engine with behavioral anomaly detection
  • WebAuthn / FIDO2 as an additional strong second factor
  • HSM/KMS-backed secrets and rotating signing keys
  • SIEM audit export for real-time SOC integration
  • Docker + orchestration for cloud/on-prem rollout
The current implementation is a proof-of-concept prototype designed to demonstrate the anti-quishing principle. The cryptographic guarantees are sound; production hardening focuses on persistence, scalability, and operational security.
